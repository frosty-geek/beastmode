## Problem Statement

Beastmode's pipeline visibility is limited to two dispatch strategies: headless SDK sessions (no visibility) and cmux terminal surfaces (requires cmux installation). Users running iTerm2 â€” the dominant macOS terminal â€” have no way to see pipeline activity rendered as native iTerm2 tabs and panes in their existing terminal workflow.

## Solution

Add `iterm2` as a third dispatch strategy that uses the `it2` CLI (the same approach Claude Code uses for its agent teams feature) to create iTerm2 tabs per epic and vertical split panes per phase within each tab. The strategy integrates into the existing `SessionFactory` pattern and extends the `auto` detection chain to prioritize iTerm2 when detected.

## User Stories

1. As a pipeline operator, I want beastmode to automatically detect that I'm running inside iTerm2 and use native tabs/panes for pipeline visibility, so that I don't need to install cmux or manually configure the dispatch strategy.

2. As a pipeline operator, I want to set `dispatch-strategy: iterm2` in config.yaml and have beastmode create one iTerm2 tab per epic with vertical split panes per dispatched phase, so that I can visually monitor parallel pipeline activity in my existing terminal.

3. As a pipeline operator, I want panes to close automatically when their phase completes and tabs to close when all panes are gone, so that the terminal stays clean without manual cleanup.

4. As a pipeline operator, I want beastmode to hard-error if I explicitly set `dispatch-strategy: iterm2` but the `it2` CLI is not installed, with clear setup instructions, so that I don't silently fall back to a different strategy.

5. As a pipeline operator using `dispatch-strategy: auto`, I want beastmode to try iTerm2 first, then cmux, then SDK â€” skipping unavailable strategies gracefully â€” so that I always get the best available visibility without manual configuration.

6. As a pipeline operator, I want iTerm2 badge notifications on errors and blocked gates only, so that I'm alerted to problems without notification spam.

7. As a pipeline operator restarting the watch loop, I want beastmode to reconcile existing iTerm2 tabs/panes from a previous run â€” adopting live ones and ignoring stale ones â€” so that I don't get orphaned terminal state.

## Implementation Decisions

- **Control mechanism**: `it2` CLI (Python-based wrapper for iTerm2's Python API). This mirrors Claude Code's ITermBackend implementation, which uses `it2 session split` for pane creation and `it2 session close` for cleanup.
- **Strategy name**: `iterm2` in `config.yaml` `cli.dispatch-strategy` (alongside `sdk`, `cmux`, `auto`)
- **Surface model**: Single iTerm2 window. One tab per active epic (named `bm-{epicSlug}`). Vertical split panes within each tab for dispatched phases/features.
- **Pane creation**: First phase in an epic gets the full tab. Subsequent phases split vertically from the existing pane using `it2 session split -v`.
- **Environment detection**: Check `ITERM_SESSION_ID` env var and `TERM_PROGRAM === "iTerm.app"` to confirm running inside iTerm2. Leader session ID read from `ITERM_SESSION_ID`.
- **`auto` strategy detection chain**: iTerm2 (if detected + it2 available) -> cmux (if available) -> SDK. This is an extension of the existing auto logic.
- **Dependency management**: `it2` is an external dependency. Beastmode does NOT auto-install it. If missing when `dispatch-strategy: iterm2` is explicitly set, beastmode prints install instructions and exits with error. In `auto` mode, missing it2 causes silent skip to next strategy.
- **iTerm2 Python API prerequisite**: Users must enable "Python API" in iTerm2 Settings > General > Magic. The setup instructions should mention this.
- **Pane lifecycle**: Panes close on phase completion via `it2 session close -f -s <paneId>`. Tab closes when last pane is removed. Mirrors worktree lifecycle.
- **Startup reconciliation**: On watch startup, query iTerm2 for existing tabs/panes matching the `bm-*` naming convention. Adopt live ones, ignore stale ones.
- **Notifications**: Use iTerm2 badge/notification system for errors and blocked gates only. Configurable via a future `iterm2.notifications` config key.
- **Completion detection**: Reuse existing `*.output.json` + `fs.watch` mechanism. Already backend-agnostic, no changes needed.
- **New TypeScript modules**: `it2-client.ts` (typed wrapper for `it2` CLI commands), `iterm2-session.ts` (ITermSessionFactory implementing SessionFactory interface)
- **SessionFactory update**: `watch-command.ts` strategy selection extended with `iterm2` case. `auto` case updated with iTerm2 priority.

## Testing Decisions

- Mock-based unit tests for `It2Client` â€” mock `Bun.spawn` / shell calls, verify correct `it2` CLI commands are constructed
- Mock-based integration tests for `ITermSessionFactory` â€” mock `It2Client`, verify session lifecycle (create tab, split pane, close pane, close tab)
- No real iTerm2 needed in CI â€” all tests run against mocked it2 responses
- Prior art: existing `cmux-client.test.ts` and `cmux-session.test.ts` follow the same mock pattern

## Out of Scope

- Auto-installation of `it2` CLI (user installs manually)
- tmux control mode (`-CC`) as an alternative to `it2`
- AppleScript-based iTerm2 control
- Pane rebalancing / tiled layouts (vertical splits only)
- Hide/show pane operations (not supported by iTerm2 API)
- Remote/SSH iTerm2 scenarios
- iTerm2 crash/restart recovery (beyond startup reconciliation)

## Further Notes

- Claude Code's ITermBackend (class `rr6` in compiled binary) is the reference implementation. It uses `it2 session split -v -s <session-id>` for pane creation and tracks panes in an array.
- Claude Code has a known bug (#41114) where backend detection sometimes selects TmuxBackend even when iTerm2 is detected. Our detection logic should be tested carefully to avoid the same issue.
- The `it2` CLI is a Python package installable via `pip install iterm2` (which provides the `it2` command). Claude Code checks for pip, pip3, pipx, and uv as package managers for installation guidance.

## Deferred Ideas

- Support for tmux control mode (-CC) as a lighter-weight alternative that doesn't require Python
- Pane tiling / layout management for high-concurrency pipelines
- iTerm2 profile customization (colors, fonts per phase type)
- Integration with iTerm2's Triggers feature for pattern-based highlighting
