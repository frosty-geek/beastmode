## Problem Statement

When a user aborts a design session (Ctrl+C), the cleanup guard in `phase.ts` never fires. The root cause is a race condition: Claude handles SIGINT internally and exits with code 0 before the `cancelled` flag is set in `factory.ts`. The dispatch wrapper reports `success: true`, the runner passes that through on its no-output early return, and `phase.ts` skips cleanup because `!pipelineResult.success` is false. This leaves orphaned worktrees, store entities, git branches, and potentially GitHub issues.

## Solution

Move design-abandon detection and cleanup from `phase.ts` into the pipeline runner's step 5 (artifact collection). When design phase produces no output.json, the runner calls `cancelEpic` directly and returns `success: false`. The `phase.ts` abandon guard is removed entirely — phase.ts becomes a pure exit-code-to-process-exit translator.

## User Stories

1. As a user, I want abandoned design sessions to clean up after themselves, so that I don't accumulate orphaned worktrees and store entries.
2. As a user, I want cleanup to fire regardless of how the design session exits (clean exit, Ctrl+C, error), so that the race condition between SIGINT and process exit doesn't matter.
3. As a developer, I want cleanup ownership in one place (the runner), so that the logic isn't split across runner and phase command with subtle success-flag dependencies.

## Implementation Decisions

- Runner step 5 (lines 283-287 in `runner.ts`): when `config.phase === "design" && !phaseOutput`, call `cancelEpic` with `force: true` and return `{ success: false, worktreePath, epicSlug }`.
- The runner will need to import `cancelEpic` from `cancel-logic.ts` and `loadConfig` from `config.ts` to obtain `githubEnabled`.
- Remove the design abandon guard from `phase.ts` (lines 142-157). Phase.ts only handles exit codes after this change.
- The runner already has `config.config` (BeastmodeConfig) available, so `githubEnabled` can be read from `config.config.github.enabled` without an additional `loadConfig` call.
- `cancelEpic` is async and warn-and-continue per step — runner failure semantics are preserved.
- The runner's early return at line 273 (`!dispatchResult.success && phase !== "validate"`) fires before step 5. If dispatch reports failure, the runner returns before reaching the no-output check. This is fine: the phase.ts guard was the only cleanup path and it was broken. For the dispatch-failure case, the runner should also check for design + no-output before its early return, or the early return should be skipped for design phase to let step 5 handle it.
- Approach: skip the early-exit at line 273 for design phase entirely. Design always falls through to step 5, which either reconciles (output exists) or cleans up (no output). This simplifies the flow to a single decision point.

## Testing Decisions

- Unit test for the runner: mock dispatch returning both `success: true` and `success: false` with no output.json present. Verify `cancelEpic` is called in both cases.
- Unit test: verify runner returns `success: false` when design has no output, regardless of dispatch result.
- Verify the phase.ts abandon guard is removed (no dead code).
- Existing cancel-logic tests remain unchanged — the cancel behavior itself is unmodified, only the call site moves.

## Out of Scope

- Simplifying the skill's epic name synthesis (separate concern)
- Removing the pre-creation slug from runner Step 0 (separate concern)
- SIGHUP/kill-9 handling (process is dead, no cleanup possible without external reaper)
- Cmux path cleanup (watch loop's ReconcilingFactory responsibility)

## Further Notes

The race condition is inherent to SIGINT handling with `stdin: "inherit"` — the child process (Claude) receives SIGINT from the terminal's process group and may exit cleanly before the parent's signal handler fires. This fix makes the race irrelevant by not depending on the dispatch exit status for cleanup decisions.
