## Problem Statement

Hook commands in `settings.local.json` use absolute paths to TypeScript files (`bun run "/Users/.../cli/src/hooks/hitl-auto.ts"`), making them brittle across machines, worktrees, and installation paths. A common failure mode is hooks silently failing because the script path doesn't exist in the current environment.

## Solution

Route all command-type hooks through the beastmode CLI as `beastmode hooks <subcommand>`, invoked via `bunx beastmode` in settings. Hook logic stays in `cli/src/hooks/` as pure library modules; a new `cli/src/commands/hooks.ts` dispatches subcommands. The pipeline runner's settings-create step generates `bunx beastmode hooks hitl-auto <phase>` instead of `bun run "/absolute/path/hitl-auto.ts" <phase>`.

## User Stories

1. As a developer on a different machine, I want hooks to resolve through the CLI binary on PATH, so that worktree sessions don't fail due to hardcoded absolute paths.
2. As the pipeline runner, I want to generate settings.local.json with `bunx beastmode hooks <name>` commands, so that hook invocations are portable across environments.
3. As a developer, I want a `beastmode hooks` subcommand that dispatches to `hitl-auto`, `hitl-log`, and `generate-output`, so that each hook has a stable CLI entry point.
4. As the CLI codebase maintainer, I want hook `.ts` files to be pure library modules without `if(import.meta.main)` entry points, so that there's a single invocation path through the CLI.
5. As the beastmode CLI, I want the `hooks` command to preserve the existing hook protocol (phase as argv, TOOL_INPUT/TOOL_OUTPUT as env vars), so that no hook logic needs to change.

## Implementation Decisions

- One CLI subcommand per hook: `beastmode hooks hitl-auto <phase>`, `beastmode hooks hitl-log <phase>`, `beastmode hooks generate-output`
- CLI invoked via `bunx beastmode` in settings.local.json — not bare `beastmode` — to handle cases where the CLI isn't globally linked
- Hook interface is unchanged: phase as positional argv, TOOL_INPUT and TOOL_OUTPUT as environment variables (Claude Code's native hook protocol)
- Prompt-type hooks (file-permission LLM evaluation) are unaffected — they're inlined in settings.local.json, not executed as commands
- New file `cli/src/commands/hooks.ts` handles subcommand dispatch, importing core logic from `cli/src/hooks/` modules
- `hooks` added as a utility command in `args.ts` alongside watch, status, cancel, etc.
- Remove `if(import.meta.main)` blocks from `hitl-auto.ts`, `hitl-log.ts`, and `generate-output.ts` — they become pure exports
- `hitl-settings.ts` builder functions (`buildPreToolUseHook`, `buildPostToolUseHook`, `buildStopHook`) updated to emit `bunx beastmode hooks <name>` instead of `bun run "<absolute-path>"`
- `file-permission-settings.ts` PostToolUse builder (`buildFilePermissionPostToolUseHooks`) similarly updated
- Plugin system remains for skill and agent distribution — this change is scoped to hooks only

## Testing Decisions

- Existing unit tests for hook logic (`hitl-auto.test.ts`, `hitl-log.test.ts`, `generate-output.test.ts`, `hitl-settings.test.ts`, `file-permission-settings.test.ts`) continue to test the core logic unchanged
- New test for `commands/hooks.ts` dispatch: verify it routes subcommands to the correct handler
- Updated tests for settings builders: assert command strings use `bunx beastmode hooks <name>` pattern instead of `bun run "<path>"`
- Integration test: verify a generated settings.local.json contains no absolute paths

## Out of Scope

- Changing the plugin system for skill/agent distribution
- Redesigning the hook argument protocol (env vars, argv shape)
- Modifying prompt-type hooks (file-permission LLM evaluation)
- Adding new hooks beyond the existing three
