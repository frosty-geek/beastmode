## Problem Statement

Claude Code's native permission dialog when writing to plugin files (.claude/, skills/, agents/, .claude-plugin/) cannot be automated or configured. Every Write/Edit operation targeting these paths triggers a manual approval prompt, blocking headless pipeline execution and requiring human intervention for routine file modifications.

## Solution

A `PreToolUse` prompt hook on Write and Edit tools with `if`-field path filtering, configured via prose in `config.yaml` under a new `file-permissions:` section organized by change category (not phase). The CLI templates the prose into hook definitions at dispatch time, using the same lifecycle as the existing HITL AskUserQuestion hooks — written at dispatch, cleaned between dispatches. A PostToolUse command hook logs all decisions to the existing HITL log for retro analysis.

## User Stories

1. As a user, I want a `file-permissions:` config section in `config.yaml` with category-based prose so that I can control how file permission dialogs are handled without per-phase configuration.
2. As a user, I want Claude Code's permission dialog for `.claude/` file writes to be interceptable by a prompt hook so that the pipeline can auto-allow or auto-deny based on my prose instructions.
3. As a user, I want file permission decisions logged alongside HITL decisions so that retro can surface patterns and suggest config evolution.

## Implementation Decisions

- **Config location**: `config.yaml` under `file-permissions:` key, sibling to `hitl:`. Organized by change category, not by phase. Phase-independent — the trust decision is about *what* you're changing, not *when*.
- **Initial categories**: `claude-settings` only, covering `.claude/**` paths. Additional categories (skills, agents, plugin-manifest) can be added later if the pattern proves useful.
- **Path patterns**: Hardcoded per category in the CLI. `claude-settings` maps to `.claude/**`. Users configure prose per category, not path globs — prevents misconfiguration.
- **Hook type**: `PreToolUse` prompt hook on Write and Edit tools. Each hook handler uses the `if` field with permission-rule syntax (e.g., `Write(.claude/**)`, `Edit(.claude/**)`). The `if` field filters *before spawning the hook process*, so there is zero LLM overhead for non-matching file paths.
- **Tools intercepted**: Write and Edit only. Bash is excluded — the `if` field matches on the command string, and there are too many ways to write files via Bash (heredoc, echo redirect, tee, etc.) for reliable interception.
- **Decision mechanism**: Prose-based LLM prompt, same pattern as AskUserQuestion HITL. The hook receives `$ARGUMENTS` (full tool input including file path and content/diff) plus the user's prose injected by the CLI.
- **Prompt context**: Full tool input via `$ARGUMENTS` — file path, content (Write), old_string/new_string (Edit). Enables content-aware decisions when the prose warrants it.
- **Three possible outcomes**: The prose can express three behaviors: (1) auto-allow — `permissionDecision: "allow"`, (2) hard deny — `permissionDecision: "deny"`, (3) defer to human — `permissionDecision: "allow"` with no `updatedInput` (silent pass-through to native dialog). This is richer than HITL which only has allow and defer.
- **Default behavior**: "always defer to human" when no prose is configured for a category. Consistent with HITL defaults.
- **Error handling**: Fail-open. Hook failure (timeout, malformed response) passes through to the native permission dialog.
- **Hook lifecycle**: Dispatch-scoped. CLI writes file-permission hooks to `.claude/settings.local.json` at dispatch time alongside HITL hooks, cleans them between dispatches. Same atomic write pattern.
- **Logging**: PostToolUse command hook appends to the same HITL log file (`.beastmode/artifacts/<phase>/hitl-log.md`). Logs tool name, file path, and decision (auto-allow, auto-deny, or deferred). Unified view with question-answering decisions.
- **Config seeding**: `beastmode init` seeds the `file-permissions:` section with `claude-settings: "always defer to human"`.
- **Hook composition**: File-permission hooks coexist with AskUserQuestion HITL hooks in `settings.local.json`. Different matchers (Write/Edit vs. AskUserQuestion), no conflict. Multiple hooks in the same event category run in parallel; most restrictive decision wins.

## Testing Decisions

- Verify CLI reads `file-permissions.claude-settings` from `config.yaml` and templates prompt hooks with `if` conditions into `settings.local.json`
- Verify prompt hook auto-allows when prose says to allow
- Verify prompt hook hard-denies when prose says to deny
- Verify prompt hook defers silently when prose says to defer or is set to "always defer to human"
- Verify `if` field correctly filters — hook only fires for `.claude/**` paths, not for other Write/Edit calls
- Verify fail-open behavior when prompt hook errors
- Verify PostToolUse log captures file permission decisions in the HITL log
- Verify `settings.local.json` cleanup between dispatches removes file-permission hooks
- Prior art: existing HITL AskUserQuestion hook tests + `hitl-settings.ts` patterns

## Out of Scope

- Additional categories beyond `claude-settings` (skills, agents, plugin-manifest)
- Bash tool interception
- Per-phase file permission configuration
- User-configurable path patterns per category
- Retro integration for file permission log analysis (retro already reads HITL logs)

## Further Notes

- The `if` field on hook handlers (permission-rule syntax) is the key enabler. Without it, every Write/Edit call would hit the LLM. With it, only matching paths trigger the hook — zero overhead otherwise.
- This design is intentionally minimal: one category, prose-based, same lifecycle as HITL. If the pattern works, adding categories is a config + path-mapping change, not an architectural one.

## Deferred Ideas

- Category expansion: skills, agents, plugin-manifest categories with their own path patterns
- Bash interception via command hook parser
- Per-phase overrides within a category
- Pattern-based categories where users define their own path groups
