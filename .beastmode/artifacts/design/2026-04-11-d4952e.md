## Problem Statement

Phase 0 (Prime) is duplicated boilerplate across all 5 phase skills. Each skill independently announces itself, loads phase context files, resolves parent artifacts by glob, and checks prerequisite gates. This duplication wastes tokens on redundant file reads, creates inconsistency between skills, prevents context caching, and makes every skill harder to maintain.

## Solution

Move all prime sub-phase logic into a single `SessionStart` hook (`bunx beastmode hooks session-start`) that runs before each Claude session. The hook reads environment variables set by the CLI, assembles phase context (L0 + L1), resolves and reads parent artifacts, evaluates gate conditions, and injects everything via `additionalContext` in the hook's JSON output. Phase 0 is removed entirely from all 5 skills — they start directly at their execute phase with context already present in the conversation.

## User Stories

1. As a skill author, I want phase context and parent artifacts injected automatically before my skill starts, so that I don't have to write context-loading boilerplate in every skill.
2. As a pipeline operator, I want the session-start hook to fail fast on missing environment variables or context files, so that Claude never starts a session with incomplete context.
3. As a pipeline operator, I want gate status (e.g., "all features implemented") injected into the conversation, so that skills can present gate failures to the user without re-computing them.
4. As a skill author, I want Phase 0 removed from all skills, so that skills are shorter, easier to maintain, and start directly with their core logic.
5. As a CLI developer, I want the session-start hook registered in `settings.local.json` alongside existing hooks, so that both the manual pipeline and watch loop paths share the same hook setup.
6. As a developer, I want unit tests covering each phase's context output shape and error paths, so that hook regressions are caught before they break sessions.

## Implementation Decisions

- **Hook communication**: Uses Claude Code's `SessionStart` hook event with `additionalContext` injection. The hook outputs JSON with `hookSpecificOutput.hookEventName = "SessionStart"` and `hookSpecificOutput.additionalContext` containing all assembled context as markdown.
- **Hook input**: Environment variables set by the CLI before dispatch: `BEASTMODE_PHASE`, `BEASTMODE_EPIC`, `BEASTMODE_FEATURE` (implement only), `BEASTMODE_SLUG`. The CLI pre-parses compound arguments (e.g., `<epic>-<feature>`) into separate env vars.
- **Implementation language**: TypeScript running via bun, consistent with existing hooks (`hitl-auto.ts`, `hitl-log.ts`, `generate-output.ts`).
- **CLI subcommand**: `bunx beastmode hooks session-start` — registered in the hooks command router alongside existing hook subcommands.
- **Context depth**: Hook injects L0 (BEASTMODE.md) and L1 (phase context header, e.g., `PLAN.md`). Skills navigate L2 convention paths themselves using the L1 index.
- **Artifact injection**: Full content injection — the hook reads parent artifact files and includes their content in `additionalContext`. Skills never read parent artifacts from disk.
- **Artifact resolution patterns**:
  - design: no parent artifacts
  - plan: `.beastmode/artifacts/design/*-$epic.md` (latest by date prefix)
  - implement: `.beastmode/artifacts/plan/*-$epic-$feature.md` (latest by date prefix)
  - validate: `.beastmode/artifacts/implement/*-$epic-*.md` + plan files for cross-reference
  - release: `.beastmode/artifacts/{design,plan,validate}/*-$epic*.md`
- **Gate evaluation**: Hook injects gate status as structured markdown sections. Skills read gate status and decide whether to stop. The hook does not block sessions on gate failure — it passes gate results through.
- **Error handling**: All errors are critical. Missing env vars, missing context files, missing required artifacts, file read errors — all cause non-zero exit, blocking session start.
- **No token budget**: No truncation or character cap. Context window handles the full injection.
- **Announcement removal**: Persona greetings removed from all phases. Design keeps its problem-first question in Execute phase (not in the hook).
- **Settings registration**: `writeSessionStartHook()` function added alongside `writeHitlSettings()` and `writeFilePermissionSettings()` in both the manual pipeline (`runner.ts` step 3) and the watch loop's dispatch path.
- **Design phase handling**: Same hook, design-aware. Injects phase context but no parent artifacts, no gates. The problem-first question moves to Design's Execute phase.

## Testing Decisions

- Unit tests for the hook logic covering each phase's context output shape
- Test cases: plan injects PLAN.md + design PRD content; implement injects IMPLEMENT.md + feature plan content; design injects DESIGN.md with no artifacts; missing env var exits non-zero; missing context file exits non-zero; missing artifact exits non-zero
- Follow existing test patterns in the CLI codebase (bun test runner)

## Out of Scope

- L2 convention path loading (skills own this, hook only provides L0 + L1)
- Topic-based context filtering (hook injects all L1 files for the phase, not topic-filtered)
- Token budget or context truncation
- Refactoring the clean/write cycle pattern for existing hooks
- Changes to how the CLI parses phase arguments (the CLI already knows phase, epic, feature)

## Further Notes

The prior art (coleam00/claude-memory-compiler `hooks/session-start.py`) demonstrates the `additionalContext` injection pattern. That implementation uses Python; this one uses TypeScript for consistency with the existing hook codebase. The core pattern is identical: read context, assemble markdown, output JSON with `additionalContext`.

## Deferred Ideas

- Refactor the clean/write cycle pattern in `hitl-settings.ts` — the `cleanHitlSettings` function may be unnecessary given that `writeHitlSettings` already replaces by matcher key
- Context caching across sessions (hash-based invalidation to skip re-reads when context files haven't changed)
