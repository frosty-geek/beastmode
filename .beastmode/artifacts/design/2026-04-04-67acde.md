## Problem Statement

The `beastmode dashboard` command hardcodes SDK dispatch (`SdkSessionFactory`) and ignores the user's configured `dispatch-strategy` in `.beastmode/config.yaml`. When the Claude Agent SDK import fails at runtime, the dashboard falls back to a broken `Bun.spawn("claude", "--print", ...)` path that cannot properly execute skills. Meanwhile, `beastmode watch` correctly reads the config and dispatches via the configured strategy (e.g., iTerm2), working as expected. This means the dashboard — the primary UI for pipeline monitoring — cannot dispatch phases at all for users whose config specifies a non-SDK strategy.

## Solution

Make the dashboard honor the configured dispatch strategy by reusing the same `selectStrategy()` function that `beastmode watch` uses. When the selected strategy doesn't support SDK streaming (iTerm2/cmux), the log panel degrades gracefully to event-based status from WatchLoop events — the same information shown by `beastmode watch`. Remove the broken `claude --print` CLI fallback from `dispatchPhase()` entirely. Add a `v` key to cycle through the 4 existing log verbosity levels (info, detail, debug, trace) at runtime for better observability.

## User Stories

1. As a pipeline operator, I want the dashboard to dispatch phases using my configured strategy (iTerm2/cmux/sdk), so that dispatch actually works instead of silently failing.

2. As a pipeline operator, I want the dashboard log panel to show event-based status (dispatching, completed, failed) when SDK streaming isn't available, so that I still have visibility into pipeline progress.

3. As a pipeline operator, I want the broken `claude --print` CLI fallback removed, so that dispatch failures surface cleanly instead of producing zombie sessions.

4. As a pipeline operator, I want to press `v` in the dashboard to cycle through log verbosity levels (info -> detail -> debug -> trace), so that I can adjust log detail at runtime without restarting.

5. As a pipeline operator, I want the dashboard to show the current log verbosity level in the key hints bar, so that I know what level I'm viewing.

6. As a pipeline operator, I want the dashboard and watch commands to use the exact same strategy selection logic, so that behavior is predictable regardless of which UI I use.

## Implementation Decisions

- Dashboard calls `selectStrategy(config.cli["dispatch-strategy"])` instead of hardcoding `SdkSessionFactory(dispatchPhase)` — same function, same priority order (iterm2 > cmux > sdk for `auto` mode)
- Dashboard wires the selected factory (ITermSessionFactory, CmuxSessionFactory, or SdkSessionFactory) into ReconcilingFactory, identical to how watchCommand does it
- When the selected strategy is not SDK, `SessionHandle.events` is undefined — the log panel falls back to rendering WatchLoop EventEmitter events (session-started, session-completed, error, release:held) instead of SDK streaming messages
- The `dispatchPhase()` function's catch block (lines 240-284 in watch.ts) that does `Bun.spawn("claude", "--print", ...)` is deleted — if SDK import fails, the error propagates through the session factory, the watch loop's processEpic catch block handles it, and the next scan cycle retries
- SDK dispatch remains a valid selectable strategy via `dispatch-strategy: sdk` — it just isn't forced anymore
- Log verbosity cycling uses a React state variable in App.tsx, starting at the verbosity passed from CLI args, cycling through 0 -> 1 -> 2 -> 3 -> 0 on each `v` keypress
- Log entry filtering happens at render time in LogPanel/TreeView — entries with level > current verbosity are hidden, but remain in the ring buffer so they appear when verbosity increases
- Key hints bar shows current level: `v verb:info` / `v verb:detail` / `v verb:debug` / `v verb:trace`

## Testing Decisions

- Unit test: dashboard creates the correct factory type for each config value (sdk, cmux, iterm2, auto)
- Unit test: `dispatchPhase()` throws when SDK import fails (no fallback path)
- Unit test: log entry filtering at each verbosity level
- Unit test: `v` key cycles verbosity state correctly (0->1->2->3->0)
- Integration test: dashboard starts and renders with each dispatch strategy
- Prior art: existing `watch-dispatch.test.ts` tests the SDK dispatch path and can be extended

## Out of Scope

- Adding SDK streaming support to iTerm2/cmux strategies (would require protocol changes)
- Per-epic or per-phase verbosity overrides
- Separate `cli.dashboard-strategy` config field
- Fixing the SDK import issue itself (may be environment-specific)

## Further Notes

- The `selectStrategy()` function already handles availability checking for iTerm2 and cmux with proper fallback chains
- The shared TreeView component between watch and dashboard means log rendering parity is nearly free
- Tree entries already store their level metadata — filtering is a render-time concern, no data model changes needed
