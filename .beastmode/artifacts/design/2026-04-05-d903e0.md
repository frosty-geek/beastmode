## Problem Statement

The HITL PreToolUse hooks for AskUserQuestion use `type: "prompt"`, spawning an LLM to interpret config prose and decide whether to auto-answer or defer. This is unnecessary overhead — the decision is binary (defer or parrot the prose) and doesn't need model inference. The prompt-based approach adds latency, cost, and a failure mode (LLM misinterpretation) to what should be a deterministic operation.

## Solution

Replace the prompt-based AskUserQuestion PreToolUse hook with a static command hook (bun TypeScript script). The script reads `.beastmode/config.yaml` at runtime, extracts the phase prose, and either:

- **Defers** (no output) when prose is "always defer to human"
- **Auto-answers** every question with answer="Other" and the prose text in `annotations[questionText].notes`

File-permission hooks (Write/Edit) stay prompt-based — they need LLM interpretation to map prose to allow/deny/defer decisions.

## User Stories

1. As a user running a design phase, I want the HITL hook to defer questions to me without invoking an LLM, so that there's no unnecessary latency or model overhead when I've configured "always defer to human."

2. As a user running an implement phase, I want the HITL hook to auto-answer AskUserQuestion calls with my config prose as freeform "Other" text, so that the agent receives my instructions without model interpretation overhead.

3. As a developer, I want the hook builder to produce `type: "command"` entries instead of `type: "prompt"` entries for AskUserQuestion, so that the settings.local.json reflects the new static behavior.

## Implementation Decisions

- **Hook type change**: `buildPreToolUseHook()` returns `type: "command"` with a bun script invocation instead of `type: "prompt"` with an LLM prompt
- **Script location**: New file `cli/src/hooks/hitl-auto.ts` — static auto-answering script, analogous to `hitl-log.ts`
- **Prose source**: Script reads config.yaml at runtime via `loadConfig()` + `getPhaseHitlProse()` — no need to bake prose into the command string
- **Phase argument**: Passed as CLI arg (same pattern as `hitl-log.ts`): `bun run hitl-auto.ts <phase>`
- **Auto-answer format**: `updatedInput.answers[questionText] = "Other"` with `updatedInput.annotations[questionText].notes = prose`
- **Defer behavior**: When prose is "always defer to human", script produces no output (silent pass-through = defer to human)
- **`buildPrompt()` removal**: The `buildPrompt()` function and all prompt template code is deleted from `hitl-settings.ts`
- **`PromptHookEntry` type update**: Changes from `type: "prompt"` to `type: "command"` in the interface
- **File-permission hooks unchanged**: `file-permission-settings.ts` stays prompt-based — LLM needed to interpret prose into allow/deny/defer
- **PostToolUse logging unchanged**: `hitl-log.ts` continues to work as-is — it already detects auto vs human from input.answers presence
- **Config schema unchanged**: The `hitl` section in config.yaml keeps the same structure; prose semantics shift from "LLM instructions" to "literal answer text" but existing values remain valid

## Testing Decisions

- Unit tests for `hitl-auto.ts`: verify defer output (no stdout) for "always defer to human", verify JSON output format for other prose
- Unit tests for `buildPreToolUseHook()`: verify it returns `type: "command"` entries targeting the new script
- Update `hitl-prompt.test.ts`: tests for `buildPrompt()` are deleted; tests for `buildPreToolUseHook()` updated to assert command type
- Update `hitl-settings.test.ts`: mock preToolUseHook entries should use command type
- Integration: existing BDD steps (`hitl.steps.ts`) should pass with the new hook format — verify decision logging still works

## Out of Scope

- File-permission hook changes (stay prompt-based)
- Config schema migration or new config format
- PostToolUse logging changes
- Retro feedback loop changes

## Further Notes

The auto-answer format (`answer="Other"` + `annotations.notes`) is a best-guess based on the AskUserQuestion schema. If Claude Code doesn't interpret this as "Other" freeform, the format will need iteration after manual testing.
