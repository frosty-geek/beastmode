## Problem Statement

Git cannot create implementation branches using the `feature/<slug>/<feature>` naming convention when the worktree branch `feature/<slug>` already exists. Git refs are filesystem paths — a ref cannot be both a leaf node and a directory simultaneously. This blocks all feature fan-out during the implement phase.

## Solution

Replace the `feature/<slug>/<feature>` convention with `impl/<slug>--<feature>`. The `impl/` prefix separates implementation branches from worktree branches in the ref namespace. The `--` separator provides an unambiguous boundary between slug and feature name. The CLI owns branch creation in `git/worktree.ts` as a single source of truth; the implement skill only verifies the branch exists.

## User Stories

1. As the CLI watch loop, I want to create `impl/<slug>--<feature>` branches before dispatching feature agents, so that fan-out dispatch never fails due to git ref conflicts.
2. As the implement skill, I want to verify I'm on the correct `impl/<slug>--<feature>` branch at prime, so that agents commit to the right branch without needing to create it.
3. As a developer running `beastmode implement <epic> <feature>` manually, I want the CLI to create the impl branch before dispatch using the same codepath as the watch loop, so that manual and automated flows behave identically.
4. As a developer rerunning implement after a failure, I want impl branch creation to be idempotent (skip if exists), so that reruns work without manual cleanup.
5. As the release phase, I want impl branches cleaned up alongside the worktree at release time, so that stale refs don't accumulate.
6. As the pipeline machine on regression, I want impl branches preserved (not deleted) when regressing to implement, so that existing work-in-progress is reusable with idempotent branch creation.

## Implementation Decisions

- Naming convention: `impl/<slug>--<feature>` replaces `feature/<slug>/<feature>`
- New `createImplBranch(slug, feature, opts)` function in `git/worktree.ts` creates the branch from the current worktree HEAD
- New `implBranchName(slug, feature)` utility in `git/worktree.ts` returns the canonical branch name for consistent usage across CLI and skill
- CLI creates the impl branch before dispatch — both watch-loop (`dispatchImplementFeatures` in `watch-loop.ts`) and manual CLI (`phase.ts`) call the same `createImplBranch()` function
- Implement skill (SKILL.md) only verifies the branch — updates the expected branch pattern from `feature/${epic}/${feature}` to `impl/${epic}--${feature}`
- Branch creation is idempotent — `createImplBranch()` checks if the branch exists and skips creation if so
- Impl branches are cleaned up at release only, alongside the worktree (no per-session cleanup)
- On REGRESS to implement, impl branches are preserved — idempotent creation handles reruns naturally
- Skill continues to own rebase at checkpoint — rebases impl branch onto the worktree branch (`feature/<slug>`)

## Testing Decisions

- Unit tests for `implBranchName()` — verifies naming convention output
- Unit tests for `createImplBranch()` — tests creation, idempotent skip, and error handling
- Integration test: create a worktree branch `feature/<slug>`, then create `impl/<slug>--<feature>` — verifies no git ref conflict
- Existing worktree tests in `__tests__/worktree.test.ts` provide patterns for git-based testing (temp repos, branch creation, cleanup)

## Out of Scope

- Impl branch rebase logic (skill-owned, already works)
- Watch-loop concurrency changes (fan-out dispatch logic unchanged, only branch creation added)
- Per-feature merge coordination (removed in current design)
- Dashboard or status display changes (impl branches are invisible to status)
