## Problem Statement

After PRD-1 (bm-d59e) delivered the structured task store (v0.84.0), the system has two sources of truth — store.json for entity hierarchy/deps/queries and per-epic .manifest.json files for phase transitions/worktree/GitHub sync. This dual-store period introduces reconciliation overhead, drift risk, and cognitive load. The manifest module (store.ts, pure.ts, reconcile.ts) and its ~55 consumer files must be rewritten to use the store as the sole persistence layer.

## Solution

Direct rewrite of ~29 production files and ~26 test files to replace all manifest reads/writes with store API calls. No adapter pattern — clean break. XState becomes the sole authority for phase transition logic, replacing manifest/pure.ts. Reconciliation logic is inlined into runner.ts. GitHub state moves to a separate sync file. A self-contained migration command (`beastmode store import`) converts existing .manifest.json files into store entities with inlined manifest parsing (no dependency on manifest/ module). After migration, manifest files and the entire manifest/ module are deleted.

This is PRD-2 of 2. Depends on PRD-1 (bm-d59e: Structured Task Store Foundation), which shipped in v0.84.0.

## User Stories

1. As a **pipeline orchestrator**, I want the pipeline machine to read epic/feature state from store.json instead of .manifest.json, so that phase transitions operate on the same data as dependency queries and ready-work computation.

2. As a **developer**, I want a single `store.json` file as the sole pipeline state, so that I don't need to understand two persistence formats or worry about drift between them.

3. As a **developer**, I want `beastmode store import` to migrate all existing manifests into the store, so that I can transition without losing pipeline state.

4. As a **watch loop**, I want to scan store entities and derive dispatch decisions from XState machine snapshots, so that epic discovery is a single JSON parse with machine-derived next actions.

5. As a **GitHub sync module**, I want to read epic/feature pipeline state from the store and GitHub refs from a separate sync file, so that pipeline state and sync state have clear ownership boundaries.

6. As a **dashboard**, I want to render epic and feature state from store entities with XState-derived enrichment, so that the TUI shows the same data as `beastmode store` commands.

7. As a **developer**, I want the manifest/ module deleted after migration, so that there is exactly one state management module in the codebase.

8. As a **pipeline runner**, I want reconciliation inlined into the 9-step execution flow reading/writing the store directly, so that post-dispatch enrichment operates on store entities without a separate reconciliation module.

## Implementation Decisions

### Three-Tier Identity Model

Every entity has three identity fields:

| Field | Scope | Mutability | Example |
|-------|-------|------------|---------|
| `id` | Internal state, store keys, git artifacts | Permanent, auto-generated | `bm-a1b2` |
| `slug` | Artifact filenames, log scopes, CLI args | Immutable once set | `manifest-absorption` |
| `name` | Display (dashboard, GitHub issue titles) | Mutable | `Manifest Absorption` |

Resolution: `store.find(idOrSlug)` accepts either id or slug. Internal state always uses id. Human-facing surfaces use slug or name.

Features also get the three-tier model: `id` (bm-xxxx.N), `slug` (from plan frontmatter), `name` (display).

### Slug Generation and Deduplication

Skills create slugs via artifact YAML frontmatter. The store normalizes and deduplicates:

1. Skill writes `slug: add-auth` in frontmatter → output.json carries it to CLI
2. Store runs `slugify()` to normalize (kebab-case, strip special chars)
3. Store checks uniqueness across all epics and features
4. On collision: append `-xxxx` suffix where xxxx is derived from the entity's id hash
5. On collision: rename the artifact file to match the new slug (`git mv`)

`slugify()` moves into the store module. `isValidSlug()` validates format.

### Git Artifact Naming

Git artifacts use the permanent entity ID:

| Artifact | Format | Example |
|----------|--------|---------|
| Branch | `feature/<id>` | `feature/bm-a1b2` |
| Phase tag | `beastmode/<id>/<phase>` | `beastmode/bm-a1b2/design` |
| Worktree path | `.claude/worktrees/<id>` | `.claude/worktrees/bm-a1b2` |
| Archive tag | `archive/<id>` | `archive/bm-a1b2` |

Artifact files use the human-readable slug: `YYYY-MM-DD-<slug>.md`

### Rename Ceremony Deletion

The 80-line `store.rename()` function is deleted entirely. With permanent ID-based git artifacts:
- Branch: `feature/bm-a3f8` (never renamed)
- Worktree: `.claude/worktrees/bm-a3f8` (never renamed)
- Tags: `beastmode/bm-a3f8/<phase>` (never renamed)
- Entity ID: `bm-a3f8` (permanent)
- Slug: immutable once set (no rename needed)
- Name: `epic.name` field (mutable, no file/branch implications)

The design phase sets `slug` and `name` when the PRD is approved. No rename of any git artifact.

### GitHub State — Separate Sync File

GitHub issue numbers and body hashes move to a separate sync file, not on store entities:

- **File**: `.beastmode/state/github-sync.json`
- **Schema**: `Record<entityId, { issue: number; bodyHash?: string }>`
- **Owner**: GitHub sync module (sole reader/writer)
- **Store stays clean**: pipeline state only, no sync concerns

The store models pipeline state; GitHub sync is a separate concern with its own persistence.

### XState Owns All Transition Logic

manifest/pure.ts is deleted entirely. XState is the sole authority for:
- Phase transitions (advancePhase, regress)
- Dispatch decisions (replaces deriveNextAction)
- Feature status tracking (replaces markFeature, shouldAdvance, getPendingFeatures)

Watch loop scan: `scanEpics()` lists store entities → hydrates XState machine snapshots → reads dispatch metadata from machine state. No separate pure functions.

### Machine Context = Epic Entity

`EpicContext` is replaced by the store's `Epic` type directly. No adapter layer:
- Machine context IS an Epic entity
- Machine creation reads Epic from store
- Machine actions write back to store
- Guards read store state directly

### Reconciliation Inlined Into Runner

manifest/reconcile.ts is deleted. Reconciliation logic (load output.json → send XState events → persist) is inlined into runner.ts steps 6 and 7. Simpler, fewer modules, fewer indirections.

### Single-Persist Pattern (Preserved)

Accumulate state changes in memory during machine transitions. Single `store.transact()` call at end of dispatch writes everything to store.json. Atomic, simple, consistent with existing pattern.

### Wave Computation

Feature waves are computed from the dependency graph via `store.computeWave(featureId)`, not stored as a field. Eliminates denormalization. Import command converts wave → depends_on relationships.

### Post-Dispatch Pipeline Changes

Current 9-step pipeline:
1. Worktree prepare → **unchanged** (uses id for worktree path)
2. Rebase → **unchanged**
3. HITL settings write → **unchanged**
4. Interactive dispatch → **unchanged**
5. Artifact collection → **unchanged** (still reads output.json)
6. Reconciliation → **rewritten**: inlined into runner, enriches store entities from output.json via XState events
7. Advance → **rewritten**: XState advances epic status in store, single transact() at end
8. GitHub sync → **rewritten**: reads pipeline state from store, sync refs from github-sync.json
9. Worktree cleanup → **unchanged**

### Migration Command

```bash
beastmode store import
```

**Self-contained** — inlines all manifest-reading code (no dependency on manifest/ module). This ensures the manifest module can be fully deleted while import remains functional.

Reads all `.beastmode/state/*.manifest.json` files and creates store entities:

1. For each manifest:
   a. Generate new `bm-xxxx` ID
   b. Create epic entity with all manifest fields mapped
   c. Set `slug` from `manifest.epic` (slugified) or `manifest.slug` (hex)
   d. Set `name` from `manifest.epic` or derive from slug
   e. Create feature entities for each `manifest.features[]` entry (with slug field)
   f. Map `manifest.artifacts` to typed fields on entities
   g. Extract `manifest.github` and `feature.github` refs into `github-sync.json`
2. Compute `depends_on` from `feature.wave` ordering (features in wave N+1 depend on all features in wave N within the same epic)
3. Validate: all entities created, all references resolve
4. Delete `.manifest.json` files on success

**Grandfathering active epics**: existing branches, tags, and worktrees keep their current slug-based names. Only new epics created after migration use id-based git artifact naming. The store tracks the actual branch/worktree names on the entity's `worktree` field regardless of naming convention.

The migration is idempotent — running it twice on the same manifests produces the same store state (existing entities with matching origin are skipped).

### Store Schema Extension

Feature type gains a `slug` field (required). Store version stays v1 (no production users yet). `slugify()` and `isValidSlug()` move into the store module.

### Backward Compatibility

Clean break after import:
- `beastmode store import` migrates everything
- `store.json` becomes sole authority immediately
- Manifest files are deleted on successful import
- No dual-read path, no fallback to manifests
- Skills continue to write artifacts with YAML frontmatter — the Stop hook generates output.json as before
- The output.json contract is unchanged — the CLI reads it and enriches store entities

- Each production file migration includes running the full test suite

- End-to-end test: design → plan → implement → validate → release cycle using store only

## Further Notes

- The blast radius is significant: ~29 production files, ~26 test files. Store tests first mitigates risk.
- GitHub sync file is a clean separation — the store models pipeline state, the sync file models GitHub refs. No compromise needed.
- The rename ceremony deletion is one of the highest-value outcomes — `store.rename()` is the most bug-prone code in the CLI.
- Wave-to-dependency conversion during import is lossy: wave ordering doesn't capture which specific features block which. The import creates conservative deps (all wave N+1 features depend on all wave N features). Manual refinement via `--add-dep` / `--rm-dep` can tighten this.
- Grandfathering active epics avoids breaking in-progress work. New epics use id-based naming; old epics keep slug-based git artifacts.
- Self-contained import command means manifest/ deletion is safe — no lingering dependencies.
