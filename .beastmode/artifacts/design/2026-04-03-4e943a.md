## Problem Statement

The CLI's log output is a flat `HH:MM:SS slug: message` string with no structure, no color, and no workflow context. When multiple epics and features are in flight, it's hard to scan which phase an epic is in, what feature a message belongs to, or distinguish warnings from info. The output looks like syslog, not a modern CLI tool.

## Solution

Rework the CLI logger to produce structured, colored, pino-pretty-inspired log lines with phase, epic, feature, and timestamp fields. Introduce a child logger API for hierarchical context propagation. Extract a shared format function used by both the CLI logger and the dashboard's activity log for visual consistency across all output surfaces.

## User Stories

1. As a developer running `beastmode plan my-epic`, I want log lines to show the phase and epic in a structured format, so that I can immediately see what phase the output belongs to.
2. As a developer watching the pipeline (`beastmode watch`), I want each log line to show phase/epic/feature context with color coding, so that I can visually distinguish between concurrent epics and features.
3. As a developer reading warnings and errors, I want them to stand out via color (yellow WARN, red ERR), so that I don't miss important messages in a stream of INFO lines.
4. As a developer piping CLI output to a file, I want ANSI colors to be automatically stripped (NO_COLOR / non-TTY detection), so that log files don't contain escape codes.
5. As a developer using the dashboard, I want the activity log panel to use the same visual format as the CLI logger, so that the output looks consistent regardless of which surface I'm reading.
6. As a developer debugging with `-vv`, I want verbose levels (DETL, DEBUG, TRACE) to be visually distinct from INFO, so that I can filter by eye when scanning output.
7. As a contributor modifying the CLI, I want the logger API to be simple (createLogger + child pattern), so that adding structured context to new code paths requires minimal boilerplate.

## Implementation Decisions

- **Log line format**: Pino-pretty style — `[HH:MM:SS] LEVEL  (phase/epic/feature):  message`
- **Column alignment**: Soft alignment — pad scope+colon to a hardcoded target column. Long scopes overflow gracefully with minimum 2-space gap before message. No truncation.
- **Level labels**: Always visible on every line. Fixed-width 5-char labels: `INFO `, `DETL `, `DEBUG`, `TRACE`, `WARN `, `ERR  `
- **Color scheme (chalk)**: Dim timestamp, green INFO/DETL, blue DEBUG/TRACE, yellow WARN (full line), red ERR (full line), cyan scope slug, dim brackets/parens
- **Color library**: chalk — added as production dependency in cli/package.json. Handles NO_COLOR, FORCE_COLOR, and isatty() automatically.
- **Logger API**: Child logger pattern. `createLogger(verbosity, { phase?, epic?, feature? })` returns a `Logger` with a `.child(partialContext)` method that returns a new Logger with merged context. Mirrors pino's `.child()` semantics without the dependency.
- **Scope model**: Always `phase/epic` or `phase/epic/feature`. No internal CLI component names (watch, post-dispatch) in the scope. Phase is the workflow phase: design, plan, implement, validate, release.
- **Fallback scope**: Call sites without phase/epic context (lockfile, github-discovery, top-level error handler) use `(cli)` as the scope label.
- **Shared format function**: `shared/log-format.ts` exports `formatLogLine()` accepting level, phase, epic, feature, message. Returns a chalk-colored ANSI string. Consumed by both `logger.ts` and dashboard React components (Ink renders ANSI strings in `<Text>` directly).
- **Null logger**: `createNullLogger()` updated to include a no-op `.child()` that returns another null logger.
- **Message cleanup**: Strip redundant epic slug prefix from watch loop subscriber messages. The scope field now carries that context.
- **Migration**: All ~15 `createLogger()` call sites migrated to the new API in a single pass. No backward-compatible overload.

## Testing Decisions

- Rewrite `logger.test.ts` for new API: createLogger with context object, child() context merging, verbosity gating (unchanged behavior), warn/error always shown (unchanged).
- New `log-format.test.ts` for the shared format function: all field combinations (phase only, phase+epic, phase+epic+feature, no context), column alignment padding, `(cli)` fallback, NO_COLOR stripping (set env var, verify no ANSI codes).
- Prior art: existing `cli/src/__tests__/logger.test.ts` uses stdout/stderr capture pattern — same approach for new tests.

## Out of Scope

- JSON/machine-readable log output mode (structured JSON lines for log aggregation)
- Log file rotation or persistence
- Per-epic or per-phase log filtering at runtime
- Custom color themes or user-configurable color schemes
- Replacing sdk-message-mapper.ts data model — only the rendering layer changes

## Further Notes

- Research artifacts saved to `.beastmode/artifacts/research/2026-04-03-structured-logging-tui-formats.md` with format comparisons from pino-pretty, stern, logfmt, turborepo, nx, and others.
- The child logger pattern was chosen over flat extended signatures based on research showing it's the consensus for CLI tools with hierarchical context (component > scope > sub-scope).

## Deferred Ideas

- Phase-colored scope (design=magenta, plan=blue, etc.) — considered but deferred in favor of uniform cyan scope for simplicity. Can revisit if scanning concurrent epics proves difficult.
- Configurable column width — hardcoded is the universal standard. Defer unless terminal width issues surface.
- JSON log mode (`--log-format=json`) for machine consumption — separate feature if pipeline log aggregation becomes a need.
