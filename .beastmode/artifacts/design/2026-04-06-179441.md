## Problem Statement

The dashboard log panel has multiple rendering bugs that together make it unusable: entries don't appear in the tree, scrolling doesn't trail, log level filtering has no effect, the layout overflows pushing the banner and keystroke hints off screen, the CLI root node breaks the hierarchical rendering pattern, unknown status badges appear for active sessions, and iTerm2 session detail is mixed into dispatch messages at the wrong log level.

## Solution

Fix 6 root causes across the dashboard's tree state building, layout sizing, node rendering, and lifecycle entry formatting. The core bug (missing `enrichedEpics` parameter) causes the tree skeleton to be empty, which cascades into broken scrolling, filtering, and refresh. The remaining fixes address layout overflow, cosmetic hierarchy, status fallbacks, and log entry formatting.

## User Stories

1. As a user watching the dashboard, I want log entries to appear in the tree panel so that I can monitor pipeline activity in real time.
2. As a user watching the dashboard, I want the log list to auto-follow new entries so that I always see the latest activity without manual scrolling.
3. As a user watching the dashboard, I want log level filtering (verbosity toggle) to hide debug entries at default verbosity so that I see a clean summary view.
4. As a user watching the dashboard, I want the banner and keystroke hints to remain visible at all times so that the layout stays stable regardless of log volume.
5. As a user watching the dashboard, I want system-level entries (watch loop events, scan results) to render as a "SYSTEM" tree node with full hierarchical formatting so that the tree is visually consistent — no exceptions.
6. As a user watching the dashboard, I want active sessions that haven't synced to the store yet to show their current phase as the status badge instead of "(unknown)" so that the tree is always informative.

## Implementation Decisions

- Pass `enrichedEpics: epics` to the `useDashboardTreeState` hook call in App.tsx — this seeds the tree skeleton so sessions have nodes to attach entries to. This single fix resolves the cascade of broken scrolling, filtering, and refresh.
- Compute `maxVisibleLines` dynamically from terminal rows: `rows - headerHeight - footerHeight - panelBorderHeight`. Pass this computed value to LogPanel instead of the hardcoded default 50. This prevents the log content from overflowing the panel and displacing the banner/hints.
- Rename the "CLI" root node to "SYSTEM" and render it as a proper hierarchical tree node. The SYSTEM node gets `│` prefix (same as epic nodes), its entries get `│ ·` leaf prefix (same as epic entries). Use Monokai muted gray for the SYSTEM node color. Update `countTreeLines`, `trimTreeToTail`, `trimTreeFromHead`, and `TreeView` to handle the SYSTEM node consistently with epic nodes.
- In `buildTreeState`, when creating dynamic epic nodes for sessions not in the store, use `session.phase` as the status instead of `"unknown"`. Same for dynamic feature nodes — use `"in-progress"` instead of `"unknown"` since a session implies active work.
- Change `lifecycleToLogEntry` for `"session-started"` to return an array of two entries instead of one: first entry at info level with `"dispatching"` message, second entry at debug level with `"session: ${id}"` detail. Update the caller in App.tsx `onSessionStarted` to push both entries.
- No change needed for the scan-complete watch loop entry — it's already at debug level and will be correctly filtered once the tree skeleton fix enables verbosity filtering.

## Testing Decisions

- Unit tests for `buildTreeState` with enrichedEpics parameter to verify skeleton seeding and entry attachment.
- Unit tests for `countTreeLines` and trim functions to verify SYSTEM node is counted consistently.
- Unit test for `lifecycleToLogEntry("session-started", ...)` returning two entries.
- Visual verification of the dashboard with the pipeline running to confirm: auto-follow works, verbosity toggle filters entries, banner and hints stay visible, SYSTEM node renders hierarchically, no unknown badges appear.

## Out of Scope

- Refactoring the scroll offset calculation (stale ref is a minor secondary issue, not user-visible once the primary fix is in place).
- Adding new log levels or verbosity tiers.
- Changing the tree hierarchy depth (CLI > Epic > Feature structure remains).
- Watch loop started/stopped entry log levels (started stays info, stopped stays debug).

## Further Notes

The enrichedEpics omission is likely a regression from a refactor that moved epic state management but didn't update all consumers. This single fix resolves 3 of the 8 reported symptoms (auto-follow, filtering, refresh).
