## Problem Statement

The beastmode plugin's Stop hook in `hooks/hooks.json` uses `$(git rev-parse --show-toplevel)` to locate hook scripts. In any non-beastmode project where the plugin is installed, this resolves to the user's project root — which doesn't contain `cli/src/hooks/generate-output.ts`. Every Claude Code response triggers a "Module not found" error. The same pattern exists in `.claude/settings.json` and in CLI-generated hooks written to `settings.local.json`.

## Solution

Remove the plugin-managed Stop hook entirely — it's redundant with what the CLI writes to `settings.local.json` per worktree. For CLI-generated hooks, replace the runtime `$(git rev-parse --show-toplevel)` shell substitution with absolute paths resolved at write time using Bun's `import.meta.dir`. This eliminates dependency on both `git rev-parse` (wrong context) and `CLAUDE_PLUGIN_ROOT` (broken upstream, Claude Code #24529).

## User Stories

1. As a plugin user working in a non-beastmode project, I want the Stop hook to not fire at all, so that I don't see "Module not found" errors on every Claude response.

2. As a beastmode CLI user, I want CLI-dispatched sessions to have correctly resolved hook script paths, so that output.json generation and HITL hooks work reliably in worktrees.

3. As a beastmode developer, I want hook path resolution to be self-contained via `import.meta.dir`, so that hooks work regardless of CWD, environment variables, or symlink configuration.

4. As a contributor, I want existing hook tests to validate absolute path resolution, so that regressions to shell-substitution patterns are caught.

## Implementation Decisions

- Remove the Stop hook entry from `hooks/hooks.json` — the file becomes an empty hooks object or is removed entirely. This eliminates the error for all plugin users in non-beastmode projects.
- Remove the Stop hook entry from `.claude/settings.json` — same redundancy reasoning. The CLI writes its own hooks to `settings.local.json`.
- In `hitl-settings.ts`, change `buildPostToolUseHook()`, `buildStopHook()`, and `buildPreToolUseHook()` to use absolute paths derived from `import.meta.dir` instead of `$(git rev-parse --show-toplevel)`. The CLI knows its own filesystem location at build time.
- In `file-permission-settings.ts`, change `buildFilePermissionPostToolUseHooks()` with the same pattern.
- The absolute path is computed once (e.g., `resolve(import.meta.dir, "../hooks/hitl-log.ts")` or similar) and embedded in the hook command string.
- Script-internal usage of `git rev-parse --show-toplevel` (in `generate-output.ts`, `hitl-auto.ts`, `hitl-log.ts`) remains unchanged — those correctly resolve the user's project root to find `.beastmode/` artifacts and config.
- `CLAUDE_PLUGIN_ROOT` is intentionally not used — it's broken upstream (Claude Code issues #24529, #32486, #27145) with no reliable fix timeline.

## Testing Decisions

- Update `hitl-settings.test.ts` assertions to expect absolute paths instead of `$(git rev-parse --show-toplevel)` patterns.
- Update `hitl-prompt.test.ts` if it asserts on the `git rev-parse` pattern.
- A simple validation: the generated hook command path should be an absolute path that exists on disk when tested from the CLI's own context.
- Prior art: existing tests in `cli/src/__tests__/hitl-settings.test.ts` already verify hook structure and command content.

## Out of Scope

- Fixing `CLAUDE_PLUGIN_ROOT` upstream in Claude Code — that's Anthropic's bug (#24529).
- Adding a SessionStart self-patcher or glob-based plugin cache resolver — over-engineering for a problem solved by removing the redundant hook.
- Changing script-internal `git rev-parse` usage — those are correct for their purpose.
