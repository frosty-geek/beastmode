## Problem Statement

The dashboard log panel has several interrelated issues that make it unreliable for monitoring pipeline activity. Logs appear twice (once flat at the CLI root, once hierarchically under the epic), the log level switcher ('v' key) has no visible effect because everything is info level and CLI entries bypass filtering, several event types use incorrect log levels (noisy heartbeats drowning out meaningful events), the iTerm session ID is dropped from dispatch log entries, and the version is not displayed anywhere in the dashboard chrome.

## Solution

Fix the dual-write pattern in App.tsx so epic-scoped events only flow to the hierarchical tree (FallbackEntryStore), stop duplicating them into systemEntries. Apply verbosity filtering to CLI root entries. Reclassify log levels so the info view shows only actionable events and debug captures operational noise. Add the version string (from WatchLoop started event) under the clock in ThreePanelLayout. Include the iTerm sessionId in the epic-scoped "dispatching" log entry.

## User Stories

1. As a pipeline operator, I want each log entry to appear only once (hierarchically under its epic), so that I can follow activity without visual noise from duplicates.
2. As a pipeline operator, I want the log level switcher ('v' key) to toggle between info and debug views, so that I can control how much detail I see.
3. As a pipeline operator, I want scan-complete heartbeats, watch loop lifecycle, session started, and session completed to be debug level, so that info view shows only actionable events.
4. As a pipeline operator, I want dead-session re-dispatch, epic-blocked, and release-held events to be warn level, so that abnormal conditions stand out.
5. As a pipeline operator, I want the iTerm session ID included in the dispatch log entry (e.g., "dispatching (session: w:12345)"), so that I can correlate dashboard entries with iTerm tabs.
6. As a pipeline operator, I want to see the current version and git hash under the clock in the top-right header (e.g., "v0.96.0 (a1b2c3d)"), so that I know which build is running.
7. As a pipeline operator, I want CLI root entries (watch start/stop, scan complete) to also respect the verbosity filter, so that switching to info actually hides debug-level global entries.

## Implementation Decisions

- **Dual-write elimination**: In App.tsx event handlers, epic-scoped events (session-started, session-completed, scan-complete with active sessions, epic-blocked, release-held, session-dead, epic-cancelled) stop calling `pushSystemEntry()`. Only truly global events (watch loop started, watch loop stopped, scan-complete when no sessions are active, errors without epicSlug) remain on the CLI root node.
- **Log level reclassification table**:
  - `scan-complete` → debug
  - `started` (watch loop) → debug
  - `stopped` (watch loop) → debug
  - `session-started` → debug
  - `session-completed` (success) → debug
  - `session-completed` (failed) → error (unchanged)
  - `session-dead` → warn (re-dispatch is abnormal)
  - `epic-blocked` → warn
  - `release:held` → warn
  - `epic-cancelled` → info (unchanged)
  - `error` → error (unchanged)
- **Verbosity filter for CLI entries**: In `filterTreeByVerbosity()` (LogPanel.tsx), apply `shouldShowEntry()` to `state.cli.entries` instead of passing them through unfiltered.
- **Version display**: Capture `version` from the WatchLoop `started` event payload in App.tsx, store in state, pass to ThreePanelLayout as a new `version` prop. Render below the clock in muted color.
- **iTerm session ID in dispatch entry**: In `lifecycleToLogEntry()` for `session-started`, include `sessionId` from the payload: `"dispatching (session: ${sessionId})"`.
- **FallbackEntryStore level propagation**: The `lifecycleToLogEntry()` function returns `type: "text"` for most events, and `entryTypeToLevel()` maps "text" to "info". To propagate the correct level, either add a `level` field to the dispatch LogEntry type, or use a type convention (e.g., "heartbeat" for debug entries). The simplest approach is to add an optional `level` field to `LogEntry` in `factory.ts` and have `entryTypeToLevel()` prefer it over the type-based mapping.

## Testing Decisions

- Pure function tests for `filterTreeByVerbosity` covering CLI entry filtering (existing tests only check epic/feature entries)
- Pure function test for `lifecycleToLogEntry` verifying session-started includes sessionId
- Integration test for `buildTreeState` verifying no duplicate entries between CLI node and epic nodes
- Existing test patterns in `cli/src/__tests__/log-panel-refactor.test.ts` and `cli/src/__tests__/tree-view.test.ts` provide prior art

## Out of Scope

- Log persistence to disk (logs are in-memory only)
- Log search/filter by text
- Changing the tree structure or adding new hierarchy levels
- Changing log colors or formatting
