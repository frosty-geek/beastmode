## Problem Statement

The Details panel in the dashboard duplicates the OverviewPanel content when "(all)" is selected and provides no operational insight about the pipeline's runtime behavior. Session timing, phase performance, and failure recovery data are all available from watch loop events but go unrendered.

## Solution

Replace the "(all)" detail card with a session stats view showing three stacked sections — session stats, phase duration breakdown, and retry stats — accumulated from watch loop events during the current dashboard session. A dedicated stats accumulator module subscribes to watch loop events and provides computed metrics to the panel.

## User Stories

1. As a user viewing the dashboard with "(all)" selected, I want to see session stats (total count, active count, success rate, uptime, cumulative session time), so that I understand pipeline throughput at a glance.

2. As a user viewing the dashboard with "(all)" selected, I want to see average phase duration broken down by phase (plan, implement, validate, release) with phase-colored names, so that I can identify slow phases and performance trends.

3. As a user viewing the dashboard with "(all)" selected, I want to see retry stats (total re-dispatches and total failures), so that I can gauge pipeline reliability during this session.

4. As a user who just started the dashboard with no completed sessions, I want to see a dim "waiting for sessions..." placeholder instead of zeros, so that the panel communicates its purpose before data arrives.

5. As a developer, I want session stats accumulated in a dedicated module (`session-stats.ts`) that subscribes to watch loop events, so that stats logic is testable and decoupled from React rendering.

## Implementation Decisions

- Scope limited to the "(all)" detail card only; epic and feature artifact views remain unchanged (raw markdown)
- Stats are current-session only — in-memory accumulation, no persistence, resets on dashboard restart
- New `session-stats.ts` module subscribes to watch loop EventEmitter events (`session-started`, `session-completed`) and accumulates counters
- Layout uses stacked sections with bold headers ("Sessions", "Phase Duration", "Retries") matching existing panel style
- Phase names in the duration breakdown use `PHASE_COLOR` from `monokai-palette.ts`; stat values rendered in default text color
- Always show all four phases (plan, implement, validate, release) in the duration breakdown; unseen phases display "--"
- Uptime updates on event-driven cadence (scan-complete events, ~60s interval), not a dedicated timer
- Both wall-clock uptime and cumulative session time shown in the Sessions section
- Empty state: dim "waiting for sessions..." placeholder until first session completes
- The `DetailsContentResult` union type gains a new `kind: "stats"` variant (or the existing `kind: "overview"` is repurposed) to carry the stats payload

## Testing Decisions

- Unit tests for `session-stats.ts`: verify counters accumulate correctly from mock events, verify average duration calculation, verify empty state detection
- Unit tests for duration formatting (seconds, minutes, hours)
- Integration test: verify DetailsPanel renders stats sections when selection is "(all)" and stats are available
- Existing tests for epic/feature artifact views must remain passing (no regressions)
- Prior art: existing `details-panel.test.ts` tests the content resolution logic; new tests follow the same pattern

## Out of Scope

- Epic detail card redesign (structured card replacing raw PRD markdown)
- Feature detail card redesign
- Persistent metrics across dashboard restarts
- Token usage or API call tracking
- Cost tracking (field exists but is never populated)
- Feature completion progress bar
- Real-time uptime tick (1-second timer)

## Further Notes

- The `costUsd` field on `SessionResult` exists but is never set in production; if it becomes populated in the future, the stats accumulator could surface it without architecture changes.
- Design phase sessions are excluded from the watch loop (interactive), so phase duration will never show a "design" row.

## Deferred Ideas

- Persistent metrics file for stats that survive dashboard restarts
- Feature completion progress indicator in the epic detail card
- Historical trend visualization (sparklines or mini-charts in TUI)
- Token/API usage tracking infrastructure
