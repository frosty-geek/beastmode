## Problem Statement

The 5 phase skill documents (design, plan, implement, validate, release) have inconsistent heading hierarchies, mixed constraint/reference placement, and overuse of headings where numbered lists would be clearer. This makes the documents harder to scan, maintain, and extend consistently.

## Solution

Restructure all 5 phase skill documents to follow a uniform convention: proper markdown hierarchy (3 levels max), flow-first section ordering, numbered lists for sequential steps, and constraints/reference material at the end.

## User Stories

1. As a skill author, I want all phase skills to follow the same document structure, so that I can maintain and extend them consistently.
2. As a skill consumer (Claude), I want a clear heading hierarchy that reflects the phase execution model, so that I can parse and follow instructions reliably.
3. As a skill author, I want guiding principles up front and constraints at the end, so that positive operating truths are internalized before encountering restrictions.
4. As a skill consumer (Claude), I want sequential steps as numbered lists instead of headings, so that I can follow the execution order without heading-level confusion.
5. As a project maintainer, I want reference material (templates, formats, deviation rules) in a dedicated section, so that phase instructions and reference data don't interleave.

## Implementation Decisions

- **Heading hierarchy**: 3 levels max — `#` for title, `##` for major sections, `###` for subsections within those
- **Phase headings**: `## Phase 0: Prime`, `## Phase 1: Execute`, `## Phase 2: Validate`, `## Phase 3: Checkpoint` — numbered prefix retained
- **Step format**: Numbered lists with bold step names. `### Subsection` headings only for complex multi-paragraph steps that need their own structure (e.g., Gray Area Sweep, Wave Loop)
- **Section order**: Title > HARD-GATE > Guiding Principles > Phase 0 > Phase 1 > Phase 2 > Phase 3 > Constraints > Reference
- **Guiding Principles**: Each skill gets 2-4 skill-specific principles extracted from current prose — not duplicates of BEASTMODE.md
- **Constraints**: All "do NOT" / "NEVER" / anti-pattern rules collected into `## Constraints` at the end
- **Reference**: Templates, formats, deviation rules, and other reference material in `## Reference` at the end, with `###` subsections
- **Reference material stays inline** — no @imports or separate files
- **Scope**: 5 phase skills only — beastmode router skill excluded (different structure is appropriate)
- **Content preservation**: All existing instructions, rules, and logic preserved — this is a structural reorganization, not a content rewrite

## Testing Decisions

- Verify each restructured skill preserves all original content (no dropped instructions)
- Verify heading hierarchy is consistent across all 5 skills (same pattern, same depth)
- Verify section ordering matches the convention in every file
- Spot-check that numbered lists are used for sequential steps and headings reserved for structural sections

## Out of Scope

- Content rewrites or new instructions — this is structural only
- The beastmode router skill (beastmode/SKILL.md)
- Subcommand files under beastmode/subcommands/
- Context files (.beastmode/context/)
- BEASTMODE.md itself

## Further Notes

- The reference skill (clone-website) demonstrates the target pattern well: guiding principles up front, clean phase flow, numbered steps, reference material inline
- Some skills have significantly more reference material than others (implement has ~150 lines of deviation rules + task format) — this is fine to keep inline under ## Reference
