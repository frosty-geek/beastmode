## Problem Statement

The identifier system is inconsistent across CLI layers. Hex slugs, human-readable slugs, and IDs compete for authority in frontmatter, env vars, store lookups, reconcile functions, and mutex locks. Seven places use seven different identifier types, creating confusion about which identifier to use where and making the codebase harder to reason about.

## Solution

Reduce to exactly 4 identifiers with clear ownership rules:

| | ID (internal) | Slug (human-facing) |
|---|---|---|
| **Epic** | `bm-7a3c` | `fix-tree-log-rendering-7a3c` |
| **Feature** | `bm-7a3c.2` | `fix-tree-log-rendering--compact-cli-7a3c.2` |

IDs for all internal use (store keys, reconcile params, mutex locks, env vars). Slugs for all human-facing use (filenames, branch names, CLI input, dashboard labels). No exceptions, no competing columns.

Design-phase epics start with a Docker-style placeholder name (`happy-bunny-7a3c`) that gets replaced at reconcile (`fix-tree-log-7a3c`). The hex suffix is immutable because the ID doesn't change.

Feature slugs embed the epic name via `--` separator: `{epic-name}--{feature-name}-{4hex}.{ordinal}`. The `--` separator is guaranteed unambiguous because `slugify()` forbids consecutive hyphens.

## User Stories

1. As a developer, I want every internal lookup (reconcile, mutex, store) to use entity IDs, so that I never wonder "is this a slug or an ID?"
2. As a developer, I want every human-facing surface (filenames, branches, CLI input, dashboard) to use slugs, so that identifiers are readable and memorable.
3. As a developer, I want feature slugs to embed their parent epic name, so that I can identify which epic a feature belongs to from its slug alone.
4. As a developer, I want a direct bijection between IDs and slugs via the hex suffix, so that I can mentally convert between them without a store lookup.
5. As a user, I want design-phase epics to have memorable placeholder names instead of hex strings, so that worktree directories and branch names are human-friendly from the start.
6. As a developer, I want frontmatter to use `id:` + `epic:` (+ `feature:`) consistently across all phases, so that artifact-to-entity matching is unambiguous.
7. As a developer, I want env vars to be `BEASTMODE_ID` + `BEASTMODE_EPIC` + `BEASTMODE_FEATURE`, so that skills always know which identifier type they're working with.
8. As a developer, I want `store.find()` removed and all internal code to use `getEpic()`/`getFeature()`, so that there's one lookup path per identifier type.
9. As a developer, I want `slugify()` deduplicated to a single import from `store/slug.ts`, so that slug formatting is consistent everywhere.
10. As a developer, I want slug validation to allow dots (for feature ID suffixes like `.2`), so that feature slugs pass validation.

## Implementation Decisions

- **Clean slate migration**: No backward compatibility for old-format store files. Existing in-progress epics must be released or cancelled before upgrading.
- **`--` separator safety**: `slugify()` collapses consecutive hyphens, making `--` in epic/feature names impossible. This guarantees unambiguous feature slug parsing.
- **Placeholder name format**: Docker-style `adj-noun` with ~50 words each (~2,500 combinations). Format: `{adj}-{noun}-{4hex}`. New file: `cli/src/store/names.ts`.
- **Branch rename timing**: Existing runner.ts rename logic (branch + worktree dir + git metadata) handles placeholder-to-real rename at design reconcile. No new rename code needed.
- **Feature slug format**: Full `{epic-name}--{feature-name}-{4hex}.{ordinal}`. Verbose but provides full context without a store lookup.
- **Context doc updates**: Deferred to retro. L0/L2 docs referencing old slug format will be updated naturally at release.
- **`store.find()` removal**: Deleted entirely. `resolveIdentifier()` handles CLI resolution via its own scanning. Internal code uses `getEpic()`/`getFeature()`.
- **Plan frontmatter omits `id:`**: Plan skill writes human-readable feature names in `feature:` field. `reconcilePlan` creates store entities with proper slugs. `generate-output.ts` matches on `fm.epic`.
- **Env var changes**: `BEASTMODE_SLUG` removed. `BEASTMODE_ID` (entity ID), `BEASTMODE_EPIC` (epic slug), `BEASTMODE_FEATURE` (feature slug, implement only) added.
- **Frontmatter contract**: `id:` + `epic:` on every artifact. `feature:` on feature-level artifacts. `slug:` field removed entirely.
- **Early epic creation**: Design phase creates the epic entity immediately (with placeholder name slug), not at reconcile. Worktree and branch use the placeholder slug from the start.
- **Artifact filename convention**: `YYYY-MM-DD-{epic-slug}.md` for epic-level, `YYYY-MM-DD-{epic-slug}-{feature-slug}.md` for feature-level.

## Testing Decisions

- Unit tests for slug validation (dots now valid), feature slug derivation (new `--` format), placeholder name generation (format, uniqueness).
- Unit tests for reconcile functions verifying ID-based params replace slug-based params.
- Unit tests for `generate-output.ts` verifying new frontmatter matching (`fm.id`, `fm.epic`, `fm.feature`).
- Integration test: design flow creates epic immediately, placeholder slug, rename preserves hex suffix.
- Grep-based verification: zero matches for `store.find(`, `randomHex`, `BEASTMODE_SLUG`, `fm.slug`, `slug:` in skills.
- Existing test patterns in `cli/src/__tests__/` provide prior art for store and reconcile testing.

## Out of Scope

- Context tree (L0/L2/L3) doc updates — deferred to retro
- Migration tooling for existing store files — clean slate, no migration
- Changes to `resolveIdentifier()` resolution chain — stays as-is (ID → slug → prefix)
- GitHub sync changes — sync uses store entities, identifier format is transparent

## Further Notes

The implementation plan in the user's spec document provides a detailed 6-wave breakdown with specific file targets per task. The plan phase should use this as the primary input for feature decomposition.

Key risk: the feature slug format change touches every phase's skill file, every reconcile function, and the artifact matching pipeline. Wave ordering must respect these dependencies — foundation (slug validation, names) before store layer before reconcile before skills.

## Deferred Ideas

- Short alias system (e.g., `@1` for last epic, `@f2` for second feature) for CLI convenience
- Slug autocomplete in shell via CLI completions
- Feature slug abbreviation for display (truncate epic name in narrow contexts)
