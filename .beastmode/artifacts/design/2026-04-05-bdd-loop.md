## Problem Statement

The beastmode workflow has an implicit BDD loop that is structurally broken across skill boundaries. Integration tests are produced as a disconnected batch feature in Wave 1, never verified per-feature during implementation, and the validate skill halts on failure instead of looping back to implement. The three testing loops that should exist — feature-level BDD verification, TDD inner loop, and cross-feature validation — are either missing or incomplete. Only the TDD inner loop (red/green/refactor per task) actually works end-to-end.

## Solution

Make the BDD loop explicit across all three skill boundaries (plan, implement, validate):

1. **Plan** produces per-feature Gherkin integration test scenarios inline in each feature plan (via batch invocation of the plan-integration-tester agent with distribution). The separate Wave 1 "integration-tests" feature is removed.

2. **Implement** gains a feature-level BDD loop: Task 0 writes the integration test from the Gherkin scenarios (starts RED), tasks execute in TDD, then the integration test is re-run (expects GREEN). On failure, targeted task re-dispatch with model escalation (max 6 retries: 2 per tier across haiku→sonnet→opus).

3. **Validate** gains a feedback loop: on integration test failure, identify the failing feature and re-dispatch it for a full re-implement cycle (max 2 re-dispatches per feature, each resets the inner retry counter).

The complete loop:

```
Design → Plan → Implement (per feature) → Validate (all features)
                    ↑                           │
                    │     FAIL (max 2/feature)   │
                    └───────────────────────────┘

Inside Implement (per feature):
  Task 0: Write integration test (RED)
  Tasks 1-N: TDD inner loop (red unit test → implement → green unit test)
  Verify: Run integration test
    GREEN → Feature done
    RED → Analyze failure → re-dispatch task → loop (max 6 retries with escalation)
```

## User Stories

1. As a workflow user, I want plan to produce Gherkin integration test scenarios inline in each feature plan, so that every feature has a clear acceptance test before implementation begins.

2. As a workflow user, I want the plan-integration-tester agent to receive all features in a batch and return grouped per-feature scenarios, so that plan can distribute them into the correct feature plans efficiently.

3. As a workflow user, I want implement to create the feature's integration test as Task 0 (from the Gherkin scenarios in the feature plan), starting in a RED state, so that I have a failing acceptance test before any implementation work begins.

4. As a workflow user, I want implement to re-run the feature's integration test after all tasks complete, so that I know whether the feature satisfies its acceptance criteria.

5. As a workflow user, I want implement to automatically analyze integration test failures, identify the responsible task, and re-dispatch it with failure context, so that broken features get fixed without manual intervention.

6. As a workflow user, I want the feature-level BDD retry loop to support model escalation (haiku→sonnet→opus, 2 attempts per tier, 6 total retries), so that stubborn failures get escalated to more capable models.

7. As a workflow user, I want validate to automatically re-dispatch failing features for a full re-implement cycle (max 2 per feature), resetting the inner retry counter each time, so that cross-feature integration failures get resolved without manual intervention.

8. As a workflow user, I want integration tests to be identified by convention and BDD framework features (tags, groups, describe blocks) rather than a separate configuration, so that the test-to-feature mapping works with any test framework.

## Implementation Decisions

- **Per-feature integration tests**: Each feature gets its own integration test, not a shared suite. Clean ownership, isolated failures.
- **Inline in feature plan**: Integration test Gherkin scenarios live in a dedicated section of the feature plan document. No separate artifact file.
- **Plan-integration-tester agent**: Kept but interface changed. Receives all features in a single batch invocation, returns grouped per-feature scenarios. Plan distributes them into feature plans.
- **Wave 1 integration feature removed**: The current dedicated "integration-tests" feature in Wave 1 is eliminated. Integration tests are now per-feature via Task 0.
- **Task 0 pattern**: The write plan always includes Task 0 as the integration test implementation. Written by the same implement-dev agent using Gherkin scenarios from the feature plan.
- **Red-first BDD**: Task 0 integration test starts RED (feature not implemented). Expected to turn GREEN after all feature tasks complete. Pure BDD — no skip/pending markers.
- **Feature-scoped verification**: At step 3.3, only the current feature's integration test runs. No regression suite against other features — that's validate's job.
- **Targeted re-dispatch**: On integration test failure at 3.3, analyze the failure output, identify the likely responsible task, re-dispatch only that task with failure context. Same escalation model applies.
- **Inner retry budget**: 6 retries at step 3.3 with model escalation (2× haiku, 2× sonnet, 2× opus).
- **Outer retry budget**: 2 full re-implement cycles per feature at validate step 4. Each resets the inner retry counter to 0, giving a fresh 6 inner retries.
- **Theoretical max**: 3 × 6 = 18 attempts per feature before hard stop.
- **Test-to-feature mapping**: Convention-based using BDD framework features (tags, groups, describe blocks). No manifest file — the framework handles it.
- **Test runner**: Uses whatever test command is already configured. Integration tests distinguished by naming convention (e.g., `*.integration.test.ts`).
- **Validate re-dispatch**: Full implement cycle (fresh write plan, all tasks). Not a targeted mini-cycle.

## Testing Decisions

- Integration tests use the project's existing test runner with naming convention for identification
- BDD framework features (tags, groups) map tests to features — the specific mechanism is a per-project testing decision
- The inner TDD loop (red unit test → implement → green unit test) is unchanged — already works
- Task 0 integration test must be runnable in isolation (feature-scoped, no cross-feature dependencies)
- Validate runs all integration tests together as a suite-level verification

## Out of Scope

- Changes to the design skill (PRD structure unchanged)
- Changes to the TDD inner loop mechanics (already working)
- Parallel feature implementation within the BDD loop (features still execute per wave ordering)
- Custom integration test configuration in config.yaml (convention-based is sufficient)
- Specific BDD framework choice (Cucumber, Vitest describe blocks, etc. — project-level decision)

## Further Notes

- The plan-integration-tester agent's prompt will need updating to produce per-feature grouped output instead of a single batch
- The implement skill's write-plan context will need a Task 0 section explaining the integration test pattern
- The validate skill needs a new "re-dispatch" capability that can invoke implement for a specific feature
- The retry/escalation budget constants should live in the skill files, not in config.yaml (they're workflow mechanics, not user preferences)

## Deferred Ideas

- Regression suite at step 3.3 (running all previously-passing integration tests alongside the current feature's test)
- Parallel feature BDD loops within a wave (currently sequential)
- Integration test coverage reporting
- Validate-to-plan feedback loop (when a feature repeatedly fails, suggest the plan needs revision)
