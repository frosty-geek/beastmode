## Problem Statement

The GitHub sync subsystem logs `WARN` messages for expected conditions during normal phase progression — missing design artifacts during implement, missing plan files during validate, idempotent branch-link retries. These dominate the log output and obscure real warnings, making the sync log unreadable during routine pipeline operation.

## Solution

Make artifact reads phase-aware using the `epic.phase` field already available in `EpicSyncInput`, downgrade misleveled log calls to their correct severity, and enrich all sync warnings with phase context for correlation.

## User Stories

1. As a pipeline operator, I want the sync to skip artifact reads when the producing phase hasn't completed, so that expected "file not found" conditions never appear in logs.
2. As a pipeline operator, I want idempotent branch-link retries and non-critical file read failures logged at `debug` level, so that `WARN` output only contains actionable issues.
3. As a pipeline operator, I want all sync warnings to include the current phase in their log context, so that I can correlate warnings with the phase that triggered them.
4. As a developer, I want a phase-ordering utility that answers "has phase X completed?" given the current phase, so that phase-aware gates are centralized and testable.

## Implementation Decisions

- Phase gate for `readPrdSections`: skip when `epic.phase === "design"` — design artifact exists from `plan` onward (written at design checkpoint)
- Phase gate for plan file reads in `syncFeature`: skip when phase is `design` or `plan` — plan artifact exists from `implement` onward (written at plan checkpoint)
- Use a phase-ordering utility (e.g., `phaseIndex` map or `isPhaseAtOrPast(current, threshold)`) to express gates declaratively — avoids string comparisons scattered through sync code
- The `Phase` type already includes the full progression: `design | plan | implement | validate | release | done | cancelled`
- Downgrade `createLinkedBranch returned null` (branch-link.ts:121) from `warn` to `debug` — null return means branch already exists, which is the idempotent success case
- Downgrade `readPrdSections: failed to read design artifact` (sync.ts:397) from `error` to `warn` — file exists but read failed is degradation not a hard error, sync continues gracefully
- Add `{ phase: epic.phase }` to the log context object for all `warn` calls in sync.ts and branch-link.ts
- Leave all 18 legitimate warn/error calls untouched — they represent actual GitHub API failures, auth issues, and retry exhaustion
- Files modified: `cli/src/github/sync.ts`, `cli/src/github/branch-link.ts`, and potentially a shared phase utility (new or existing)

## Testing Decisions

- Unit test the phase-ordering utility: `isPhaseAtOrPast("design", "plan")` returns false, `isPhaseAtOrPast("implement", "plan")` returns true, edge cases for `done` and `cancelled`
- Integration-level verification: run sync with a design-phase epic and confirm no WARN output for missing artifacts
- Existing test patterns in `cli/src/__tests__/` use vitest with bun runtime

## Out of Scope

- Changing the warn-and-continue pattern for GitHub API failures — that's working correctly
- Adding structured log levels or log filtering infrastructure
- Modifying the reconcile.ts retry queue logging — those warnings are for actual persistent failures
- Early-issues.ts logging — all warnings there are for actual API failures

## Further Notes

The audit found 20 warn/error calls across 6 files in `cli/src/github/`. Only 4 are misleveled. The remaining 16 correctly signal actual degradation or errors.
