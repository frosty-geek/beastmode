## Problem Statement

The `beastmode watch` log output has broken visual alignment because variable-length scope names (e.g., `(cli):` vs `(implement/dashboard-rework/epics-panel):`) cause ragged indentation. Log messages also duplicate information already present in the scope — phase and feature names appear in both the structured prefix and the message text. Additionally, `costUsd.toFixed()` crashes when session errors don't include cost data.

## Solution

Restructure the log format by extracting phase as a fixed-width column, truncating epic/feature names to a 32-character budget with trailing ellipsis, and systematically removing duplicate information from message text across all log call sites. Guard against undefined costUsd.

## User Stories

1. As an operator watching the pipeline, I want log lines to align vertically so that I can scan output quickly without visual noise from ragged indentation.
2. As an operator watching the pipeline, I want to see the phase as a distinct visual column so that I can filter by phase at a glance without parsing the scope path.
3. As an operator watching the pipeline, I want log messages to contain only new information (not repeat what's in the scope and phase columns) so that each line is concise.
4. As an operator watching the pipeline, I want the watch loop to not crash on `costUsd.toFixed()` when cost data is missing so that session errors are reported cleanly.
5. As a dashboard user, I want the dashboard activity log to use the same improved format so that the headless and TUI experiences are consistent.

## Implementation Decisions

- Phase is extracted as its own fixed-width column (9 characters, padded to "implement" width). When no phase is present, the column pads with spaces to preserve alignment.
- Scope contains only epic and optional feature — no longer includes phase. Falls back to "cli" when no epic is set.
- Scope has a 32-character total budget. When only epic is present, it gets the full 32. When both epic and feature are present, each gets 16. Names exceeding their budget are truncated with trailing ellipsis.
- Message text at all log call sites is deduped: phase, epic slug, and feature slug are stripped from message strings when the logger scope already contains them. This applies to `watch.ts` (dispatching, completion) and `post-dispatch.ts` (failure, skip, reconcile, rename messages).
- Dispatching messages become just "dispatching" — the phase column and scope already identify what's being dispatched.
- costUsd is guarded: when undefined, the cost portion is omitted from the completion message entirely (not defaulted to $0.00).
- Error/warn lines continue to color the entire line (unchanged behavior, now including the new phase column).
- The format change propagates to the dashboard activity log automatically via the shared `formatLogLine` function.
- No-phase lines (startup, shutdown, generic errors) show blank phase column with "cli" in scope, spaces preserve column alignment.

## Testing Decisions

- Unit tests for `formatLogLine` covering: phase extraction, scope truncation at boundary (exactly 16, exactly 32), ellipsis behavior, no-phase padding, error/warn full-line coloring with new phase column.
- Existing logger tests in `cli/src/__tests__/logger.test.ts` provide prior art for the test pattern (capture stdout/stderr writes, assert format).
- Snapshot or visual assertion of sample log lines to catch alignment regressions.

## Out of Scope

- Log level filtering changes
- Structured JSON log output
- Log file rotation or persistence
- Changing the color scheme
- Changes to the dashboard layout beyond format propagation
