## Problem Statement

Beastmode has three disconnected version sources: a hardcoded `"0.1.0"` in the CLI help text, `cli/package.json` read by the watch loop/dashboard, and `.claude-plugin/plugin.json` read by the npx CLI. Only plugin.json is bumped during releases, so the CLI permanently claims to be v0.1.0 and the dashboard shows whatever stale value is in cli/package.json. The tool doesn't know its own version.

## Solution

Consolidate all version reads to `.claude-plugin/plugin.json` as the single source of truth. Create a shared version module in the CLI that discovers plugin.json relative to its own file location (works in both repo and plugin cache contexts). Remove the hardcoded VERSION constant. Drop git hash from the display format since it's meaningless when beastmode is installed as a plugin into another project. Display format: `v{semver}`.

## User Stories

1. As a user running `beastmode help`, I want to see the actual release version (e.g., `v0.102.0`), so that I know which version I'm running.
2. As a user viewing the dashboard, I want the version display to reflect the real plugin version from plugin.json, so that the dashboard doesn't show a stale or incorrect version.
3. As a developer, I want a single shared version module (`cli/src/version.ts`) that resolves the version from plugin.json via `import.meta.dirname` relative path traversal, so that version logic is DRY and testable.
4. As a developer, I want the version module to gracefully return `"unknown"` when plugin.json is unreadable, so that missing files don't crash the CLI or dashboard.

## Implementation Decisions

- **Single source of truth**: `.claude-plugin/plugin.json` — the release skill already bumps this file; no other version file needs updating.
- **Shared version module**: New `cli/src/version.ts` exporting a `resolveVersion(projectRoot?: string)` function. Uses `import.meta.dirname` to navigate to `.claude-plugin/plugin.json` relative to the module's own location. This works in both the repo checkout and the plugin cache because the directory structure is identical.
- **CLI help**: Remove hardcoded `VERSION = "0.1.0"` from `cli/src/index.ts`. Import from shared version module instead.
- **Watch loop**: Replace `resolveVersion()` in `cli/src/commands/watch-loop.ts` that reads `cli/package.json` + git hash with an import from the shared version module.
- **Display format**: `v{semver}` only (e.g., `v0.102.0`). No git hash — when installed as a plugin, `git rev-parse` returns the host project's hash, not beastmode's. The hash was always a lie outside the beastmode repo.
- **Fallback**: Return `"unknown"` if plugin.json cannot be read. Never crash.
- **npx CLI**: Leave `src/npx-cli/version.mjs` unchanged — it already reads plugin.json correctly and runs on a different runtime (Node.js ESM vs Bun).
- **Skills**: Not version-aware. Skills are markdown prompts — version display is a CLI/dashboard concern.
- **package.json files**: Not synchronized. `package.json` (root) and `cli/package.json` version fields become irrelevant. plugin.json is the only authority.

## Testing Decisions

- Unit test for the shared version module: mock `readFileSync` to verify it reads plugin.json and returns the formatted version string.
- Unit test for fallback: verify `"unknown"` is returned when plugin.json doesn't exist or is malformed.
- Existing dashboard version display tests (`cli/features/version-display.feature`, `cli/src/__tests__/version-display.test.ts`) should be updated to expect the new format (no git hash).
- Integration: `beastmode help` output includes a valid semver version string.

## Out of Scope

- Synchronizing version across package.json files — plugin.json is the only authority.
- Skill version awareness — skills are prompts, not versioned binaries.
- Git hash in version display — meaningless outside the beastmode repo.
- Build step or code generation — runtime file read is sufficient.
