## Problem Statement

The integration test suite grows monotonically — every epic produces new scenarios regardless of whether it changes user-facing behavior, and there is no mechanism to merge overlapping scenarios or remove stale ones. At 151 scenarios across 52 feature files, the suite has outgrown its purpose as a high-level behavioral verification layer.

## Solution

Add a behavioral-change skip gate in the plan skill that filters out non-behavioral features before dispatching the integration-tester agent, and upgrade the agent itself to perform inline suite consolidation (merging overlapping scenarios, removing stale ones) alongside new scenario generation. Shift scenario organization from per-feature 1:1 mapping to capability-based grouping with happy-path + risk-based test depth, respecting the test pyramid.

## User Stories

1. As a plan skill, I want to evaluate each feature's behavioral impact before dispatching the integration-tester agent, so that documentation-only changes, refactoring, configuration changes, and fixes with existing coverage skip integration test generation entirely.

2. As a plan-integration-tester agent, I want to review the full existing test suite for overlapping and stale scenarios on every run, so that the suite stays consolidated without requiring a separate periodic cleanup pass.

3. As a plan-integration-tester agent, I want to organize scenarios by capability domain rather than by originating feature, so that the test suite gives a holistic picture of what beastmode does instead of mirroring the feature decomposition tree.

4. As a plan-integration-tester agent, I want to prioritize happy-path scenarios and include error paths only when they represent high-risk user-visible behavior, so that the integration suite tests at the right level of the test pyramid.

5. As a plan-integration-tester agent, I want to produce a two-section artifact (New Scenarios + Consolidation) with both epic and capability tags, so that changes to existing scenarios are tracked alongside new additions in a single auditable document.

## Implementation Decisions

- Skip gate lives in the plan skill, before agent dispatch. The agent is never spawned for skipped epics.
- Skip criteria: documentation-only, refactoring/code cleanup, configuration changes, bug fixes with existing test coverage. Classification uses a behavioral-change heuristic evaluated from PRD content and feature descriptions — no explicit field in the PRD.
- Skip operates per-epic with feature filtering: if all features match skip criteria, the entire agent dispatch is skipped. If some features need tests, only those features are sent to the agent.
- When the skip gate fires for the entire epic, no integration artifact is produced. Feature plans get empty Integration Test Scenarios sections.
- Consolidation runs inline with every agent invocation — not a separate periodic pass.
- Consolidation actions: merge overlapping scenarios (same behavioral intent) and remove stale scenarios (obsolete features). No demotion tracking or hard scenario caps.
- The agent's consolidation decisions are authoritative — no human review step for merges and removals.
- Artifact restructured from three sections (New/Modified/Deleted) to two sections: New Scenarios + Consolidation (absorbs modified, deleted, and merged changes to existing scenarios).
- Scenario organization shifts from per-feature grouping to capability-based grouping (e.g., @pipeline, @dashboard, @release). Scenarios can span multiple features.
- Tagging uses both epic tags (@<epic-name>) and capability tags (@<capability-domain>). Epic tags provide traceability; capability tags provide logical grouping.
- Test depth follows happy paths + risk-based selection. Every capability gets at least one happy path scenario. Error paths included only when the agent judges them high-risk based on complexity. No edge cases at integration level.
- Changes land in two files: plan skill (skip gate logic) and plan-integration-tester agent definition (consolidation, capability grouping, artifact restructuring, test depth guidance).

## Testing Decisions

- The plan skill's skip gate is testable via existing integration test patterns: Given an epic with only documentation features, When the plan phase runs, Then no integration artifact is produced.
- Consolidation behavior is testable by providing the agent with a known set of overlapping/stale scenarios and verifying the output artifact.
- Prior art: existing `cli/features/` Cucumber tests demonstrate the declarative Gherkin style. The plan-integration-tester agent already follows this pattern.

## Out of Scope

- Automatic scenario budget or hard cap on scenario count
- Demotion of integration scenarios to unit test level
- Separate periodic consolidation agent or skill
- Changes to the validate phase's test execution
- Changes to step definitions or test infrastructure
- Retroactive one-time audit of the existing 151 scenarios (the first consolidation run will handle this organically)
