## Problem Statement

GitHub epic issue bodies are empty shells — missing all six PRD sections (Problem Statement, Solution, User Stories, Implementation Decisions, Testing Decisions, Out of Scope). Feature issue bodies lack plan content (User Stories, What to Build, Acceptance Criteria). The root cause: code that assumes non-worktree filesystem layout. The design artifact path stored in the epic entity is an absolute worktree path (e.g., `/.../.claude/worktrees/ed09f0/.beastmode/artifacts/design/2026-04-05-ed09f0.md`) that becomes invalid the moment the worktree is renamed or cleaned up. Feature plan paths are bare filenames with no directory prefix, so `resolve(projectRoot, feature.plan)` resolves to the wrong location. The `buildOutput` function stores absolute paths for design, validate, and release phases — all three become stale after worktree rename.

## Solution

Localized fixes with no shared infrastructure. Readers extract `basename()` from whatever path is stored and prefix the known artifact directory (`readPrdSections` prefixes `.beastmode/artifacts/design/`, feature plan reader prefixes `.beastmode/artifacts/plan/`). The write path in `buildOutput` stores bare filenames via `basename(artifactPath)` for all three path-storing phase cases (design, validate, release). The `buildArtifactsMap` display function uses the same `basename()` + known directory prefix approach for clean artifact table rendering. No store migration — the `basename()` approach tolerates absolute paths, relative paths, and bare filenames equally. Explicit debug logging added throughout the GitHub sync path to surface path resolution, file reads, and errors.

## User Stories

1. As a project observer, I want epic issue bodies on GitHub to contain the full PRD (Problem Statement, Solution, User Stories, Implementation Decisions, Testing Decisions, Out of Scope), so that the entire design is readable without leaving GitHub.
2. As a project observer, I want feature issue bodies on GitHub to contain the full feature plan (User Stories, What to Build, Acceptance Criteria), so that each feature's scope is visible in GitHub.
3. As a developer, I want the commit-issue-ref pipeline step to succeed in worktrees, so that commits are amended with issue references and appear in the GitHub issue timeline. (NOTE: Already fixed in current code — `amendCommitsInRange` uses `git rev-parse --git-dir`. Retained for completeness.)
4. As a project observer, I want the Artifacts table in epic bodies to show clean repo-relative paths (e.g., `.beastmode/artifacts/design/...`) instead of stale worktree paths, so that artifact links are meaningful.
5. As a developer, I want explicit debug logging in the GitHub sync path, so that path resolution failures and file read errors are visible without attaching a debugger.

## Implementation Decisions

- **Path contract**: Store bare filenames in the epic/feature store fields. Readers prefix the known artifact directory at read time. This eliminates all path format assumptions — `basename()` extracts the filename from any stored format (absolute, relative, bare).
- **Design artifact read path**: `readPrdSections` in `sync.ts` extracts `basename(designPaths[0])` and resolves against `join(projectRoot, ".beastmode", "artifacts", "design")` instead of using the stored path directly. Handles existing absolute worktree paths and future bare filenames equally.
- **Feature plan read path**: The feature plan reader in `syncFeature` extracts `basename(feature.plan)` and resolves against `join(projectRoot, ".beastmode", "artifacts", "plan")` instead of `resolve(projectRoot, feature.plan)`. Existing bare filenames already work; this adds the missing directory prefix.
- **Design artifact write path**: `buildOutput` in `generate-output.ts` stores `basename(artifactPath)` instead of the full absolute `artifactPath` for the design phase case.
- **Validate artifact write path**: `buildOutput` stores `basename(artifactPath)` for `artifacts.report` in the validate phase case. Same pattern as design.
- **Release artifact write path**: `buildOutput` stores `basename(artifactPath)` for `artifacts.changelog` in the release phase case. Same pattern as design.
- **Plan artifact write path**: `scanPlanFeatures` already stores bare filenames (`basename(filePath, ".md") + ".md"`). No change needed.
- **Artifact links table**: `buildArtifactsMap` in `sync.ts` applies `basename()` + known directory prefix per phase instead of the current `relative(projectRoot, rawPath)` approach. Produces clean display paths like `.beastmode/artifacts/design/filename.md`.
- **Commit-issue-ref git directory**: Already fixed — `amendCommitsInRange` uses `git rev-parse --git-dir` with fallback to `join(cwd, ".git")`. No changes needed.
- **Debug logging**: Add explicit logging throughout the GitHub sync path using the existing Logger (`opts.logger`) with `info`, `debug`, `warn`, `error` levels. Log path resolution steps (stored value, basename, resolved absolute path), file existence checks, section extraction results, and surface errors with full context. Use structured data parameter for machine-readable details.
- **No store migration**: The `basename()` approach is a universal adapter — it produces the correct filename regardless of what format the store contains. Old absolute paths, old bare filenames, and new bare filenames all yield the same basename.
- **No normalization function**: Each reader knows its own artifact directory. No shared utility needed. Localized fixes in localized files.

## Further Notes

The first sync after design completion (pipeline step 8) also fails — not just subsequent syncs. Step 7 renames the worktree from the hex slug to the epic name before step 8 runs, so the absolute path stored during step 6 reconciliation is already stale by the time the sync reads it. This means no epic has ever had PRD content in its GitHub body.

The `basename()` approach works because artifact filenames are globally unique (date-prefixed, slug-scoped) and each artifact type has exactly one known directory. Design artifacts always live in `.beastmode/artifacts/design/`, plan artifacts in `.beastmode/artifacts/plan/`.

The commit-issue-ref fix (User Story #3) was found to be already implemented during design-phase code exploration. Retained in the PRD for traceability.
